## Goal

Migrate `apps/api/src/durable-objects/commands-do.ts` from a plain `DurableObject<Env>` to an Agent-native command registry where command definitions, values, and counters are persisted as serializable runtime data instead of being tied to SQL seed migrations or hard-coded catalogs.

## Final design decisions

### Source of truth

- **Agent state is authoritative** for command definitions, values, and counters.
- Legacy SQLite tables are used **once** in `onStart()` to import existing production data.
- There is **no code-defined default catalog/versioning system** anymore.

### Persisted command model

Each command definition is now persisted in Agent state with serializable metadata:

- `name`
- `description`
- `category`
- `responseType`
- `permission`
- `enabled`
- `createdAt`
- `aliases`
- `valueSourceName`
- `counterSourceName`
- `handlerKey`
- `outputTemplate`
- `emptyResponse`
- `writePermission`

This lets new commands be created/updated/deleted at runtime without schema migrations.

### Stored vs computed commands

- **Stored commands** (`static` / `dynamic`) read/write persisted values.
- **Computed commands** persist a `handlerKey` string so the registry is data-driven while the live execution logic still resolves to code at runtime.
- `project` and `today` remain separate stored command definitions, but both share the same persisted value via `valueSourceName: "today"`.

### Removal behavior

- Deleting a command hard-removes it from Agent state.
- Dependent commands that share its `valueSourceName` / `counterSourceName` are deleted too (e.g. deleting `today` also removes `project`).

## Implemented

### `apps/api/src/durable-objects/commands-do.ts`

- Converted class to `Agent<Env, CommandsAgentState>`.
- Added `initialState` and `onStart()`.
- Added one-time legacy import from Drizzle tables.
- Added `validateStateChange(...)` invariants.
- Added `onStateChanged(...)` logging.
- Refactored reads/writes to use Agent state only after import.
- Added runtime CRUD RPC methods:
  - `createCommand(...)`
  - `updateCommand(...)`
  - `deleteCommand(...)`
- Preserved existing RPC methods used by callers.

### `apps/api/src/routes/webhooks.ts`

- Removed the hard-coded command allowlist.
- Chat command dispatch now looks up command definitions from `COMMANDS_DO` and routes dynamically.
- Stored commands now render using persisted command metadata (`outputTemplate`, `emptyResponse`, `writePermission`).
- Computed commands dispatch through persisted `handlerKey` values.
- `!update` permission checks now use persisted `writePermission` instead of special-casing command names.

### `apps/api/src/lib/analytics.ts`

- Widened chat command metric type from a fixed union to `string` so runtime-created commands can be tracked.

### `apps/api/wrangler.test.jsonc`

- Added the missing `COMMANDS_DO` binding and test migration entry.

### Tests

Added `apps/api/src/__tests__/durable-objects/commands-do.test.ts` covering:

- legacy hydration into Agent state
- runtime create/update/delete
- dependent delete behavior (`today` / `project`)
- computed command counter persistence

## Follow-up notes

- Runtime CRUD now exists at the DO RPC layer.
- I did **not** add a separate moderator-facing command-management UX/API surface in this task; chat invocation is now data-driven, but create/delete workflows still need a chosen caller surface if you want mods to manage definitions directly from chat or HTTP.
