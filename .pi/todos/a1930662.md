## Summary

Migrate `apps/api/src/durable-objects/event-bus-do.ts` from `DurableObject<Env>` to `Agent<Env, EventBusAgentState>`.

**Chosen architecture:** a **schedule-only Agent design** (not queue-backed, not workflow-backed).

Keep `pending_events` and `dead_letter_queue` in SQLite as the durable/admin-queryable source of truth. Use Agent state only for small operational coordination data around active delayed schedules.

Preserve the current public RPC/admin contract unless explicitly called out below.

---

## 1. Agent API inventory and relevance analysis

### Immediately useful for EventBus

#### Lifecycle / state
- `initialState`
  - Use for default operational coordination state only.
- `this.state`
  - Read the currently tracked retry-sweep / DLQ-purge scheduling metadata.
- `setState()`
  - Persist small schedule-coordination updates.
- `onStart()`
  - Required bootstrap point for migrations, legacy alarm cleanup, and schedule restoration.

#### Scheduling / coordination
- `schedule()`
  - Primary replacement for raw `alarm()`.
  - Use for one-shot delayed retry sweep wakeups and one-shot delayed DLQ purge wakeups.
- `getSchedule()` / `getSchedules()`
  - Use to reconcile stored schedule ids against actual scheduled tasks on startup/tests.
- `cancelSchedule()`
  - Use when pending/DLQ state changes invalidate the currently tracked wakeup.

### Useful later, but should not shape this migration

#### State hooks
- `validateStateChange()`
  - Optional guardrail if we want to enforce schedule-id / timestamp invariants.
  - Not required for the first migration.
- `onStateChanged()`
  - Low-value here because EventBus has no client-sync surface; avoid using it for core logic.

#### Error / long-running helpers
- `onError()`
  - Could centralize Agent callback error logging later, but core EventBus delivery already wraps/logs failures explicitly.
- `retry()`
  - Useful for short in-callback retries against transient external failures, but **do not** use it to replace domain retry semantics (`1s`, `4s`, `16s`) already modeled in SQLite.
- `keepAlive()` / `keepAliveWhile()`
  - Potentially useful later if a retry sweep starts processing very large due batches or long handler calls.
  - Not required for the current single-handler, small-batch design.
- `destroy()`
  - Not useful for singleton EventBus lifecycle.

#### Queue APIs (evaluated, rejected for this migration)
- `queue()`
- `dequeue()`
- `dequeueAll()`
- `dequeueAllByCallback()`
- `getQueue()`
- `getQueues()`

Why not now:
- EventBus ordering is driven by **`nextRetryAt`**, not FIFO insertion order.
- Queue retries are head-of-line blocking and have **no built-in DLQ**.
- Pending/DLQ already need SQLite for admin inspection.
- Queue would add an extra execution layer without removing the core need for DB-backed due-time scheduling.

### Workflow APIs: useful later, not for this bus migration
- `runWorkflow()`
- `sendWorkflowEvent()`
- `approveWorkflow()` / `rejectWorkflow()`
- `pauseWorkflow()` / `resumeWorkflow()` / `restartWorkflow()` / `terminateWorkflow()`
- `getWorkflow()` / `getWorkflows()` / `getWorkflowStatus()`
- `onWorkflowProgress()`
- `onWorkflowComplete()`
- `onWorkflowError()`
- `onWorkflowEvent()`
- `onWorkflowCallback()`
- `AgentWorkflow` concepts

Decision:
- Workflows are **better reserved for saga replacement / richer orchestration**, not a single-handler event bus with 3 attempts and a small DLQ.
- Using workflows here would duplicate retry/audit state already modeled in `pending_events`/`dead_letter_queue` and make admin/debug compatibility harder.

### PartyServer / inherited server APIs

#### Operationally relevant but not part of the EventBus design
- `name`
- `setName()`
  - Important for tests; some Agent test setups must call `await stub.setName("event-bus")` before exercising state/scheduling.

#### Intentionally out of scope for EventBus
- `broadcast()`
- `getConnection()`
- `getConnections()`
- `getConnectionTags()`
- `onConnect()`
- `onMessage()`
- `onClose()`
- `onRequest()`
  - EventBus is internal RPC/admin-route driven; no direct Agent HTTP/WebSocket surface is needed for the migration.

### Explicitly rejected for this migration unless a concrete EventBus need appears
- browser/client `@callable()` APIs
  - EventBus is DO-to-DO RPC only.
- websocket state sync / protocol messages
  - No browser-facing live state.
- email APIs
  - No email use case.
- MCP APIs
  - Not relevant to internal event routing.
- sub-agents / facets
  - Overkill for a singleton internal router.
- AI scheduling helpers like `scheduleSchema` / `getSchedulePrompt`
  - EventBus retry timing is deterministic infrastructure, not natural-language scheduling.

### Responsibility mapping
- **Agent state**
  - active retry sweep schedule id / due timestamp
  - active DLQ purge schedule id / due timestamp
- **SQLite**
  - pending delivery rows
  - DLQ rows
  - attempts / timestamps / error text
  - admin/debug query surface
- **Schedule**
  - next retry sweep wakeup
  - next DLQ purge wakeup
- **Workflow**
  - future-only if EventBus ever becomes true orchestration instead of simple routed delivery

## 2. Current EventBus contract inventory

### Current public RPC/admin contract to preserve
- `publish(event)`
- `getPendingCount()`
- `getPending({ limit, offset })`
- `getDLQ({ limit, offset })`
- `replayDLQ(id)`
- `deleteDLQ(id)`

### Current observable behavior
- `publish()` validates synchronously with Zod.
- `publish()` looks up routing synchronously via `EVENT_ROUTES`.
- `publish()` attempts immediate delivery inline.
- If the initial delivery fails, the event is persisted for retry and `publish()` still returns success.
- Retry backoff is currently `1s`, `4s`, `16s`.
- Max attempts is currently `3` total delivery attempts.
- After exhaustion, the event moves to DLQ.
- DLQ rows expire after `30` days.
- Admin routes expose pending/DLQ inspection plus replay/delete.

### Current alarm responsibilities
`alarm()` currently does four jobs:
1. purge expired DLQ rows
2. load due pending rows
3. retry delivery / move exhausted rows to DLQ
4. reschedule the next alarm

### Persistence boundaries already in place
Keep these boundaries:
- `pending_events` = retry queue / retry state
- `dead_letter_queue` = exhausted failures / admin surface
- downstream idempotency (`event_history`) stays in `AchievementsDO`
- event schema validation stays in Zod schema module
- routing stays static in code via `EVENT_ROUTES`

### Important current nuance
`EventBusDO` does **not** currently track successful delivery history. Already-delivered duplicate event IDs are effectively handled by downstream idempotency (`AchievementsDO.recordToEventHistory()` with `onConflictDoNothing`). Preserve that boundary unless there is a compelling new need.

## 3. Architecture comparison and recommendation

### Candidate A — Minimal Agent port (state + schedule, no queue)

#### Shape
- `EventBusDO` becomes `Agent<Env, EventBusAgentState>`.
- Keep existing DB tables.
- Replace `alarm()` with one-shot delayed `schedule()` callbacks.
- Maintain one retry-sweep wakeup for the earliest due pending row.
- Maintain one purge wakeup for the earliest DLQ expiry.

#### Pros
- Lowest migration risk.
- Best behavioral parity with the current alarm model.
- Fits due-time-driven semantics better than FIFO queueing.
- Preserves admin/debug query model cleanly.
- No new workflow binding/config churn.

#### Cons
- Still keeps a small amount of bespoke retry-orchestration logic.
- Retry tick still queries due rows from SQLite.

### Candidate B — Queue + schedule hybrid

#### Shape evaluated
- SQLite remains source of truth.
- `queue()` would handle immediate/asynchronous delivery units.
- `schedule()` would wake the Agent when delayed retries become due.

#### Pros
- Could decouple some work from `publish()` latency.
- Might help later if routing fan-out grows.

#### Cons / why rejected
- Queue ordering is FIFO, but EventBus semantics are driven by `nextRetryAt`.
- Queue retries are head-of-line blocking.
- Queue has no DLQ/admin surface, so SQLite still carries the real state.
- Adds another persistence/execution layer without meaningfully simplifying current EventBus logic.
- Inline delivery is already desirable for current callers and behavioral parity.

### Candidate C — Workflow-backed delivery orchestration

#### Pros
- Strong durable execution model.
- Natural fit for long-running multi-step pipelines.
- Better if EventBus ever becomes orchestration-heavy.

#### Cons / why rejected
- Overkill for a single-handler, 3-attempt backoff loop.
- Duplicates state that already exists in pending/DLQ tables.
- Makes current admin/debug contract harder to preserve.
- Better investment target for saga replacement, not EventBus.

### Final recommendation
**Choose Candidate A: schedule-only Agent design.**

Why this best matches the repo:
- EventBus is an internal singleton router, not a user-facing workflow engine.
- The durable/queryable/admin-facing state is already correctly modeled in SQLite.
- The main thing Agent buys us here is **bootstrap in `onStart()` and agent-managed delayed wakeups**, not queue/workflow abstraction for its own sake.
- The queue adds little for a due-time-driven bus, and workflows are heavier than the problem requires.

## 4. Target state model and persistence split

### Proposed Agent state

```ts
interface EventBusAgentState {
  retrySweepScheduleId: string | null;
  retrySweepDueAt: string | null;
  dlqPurgeScheduleId: string | null;
  dlqPurgeDueAt: string | null;
}
```

### State ownership rules

#### Agent state
Only small operational coordination data:
- current retry-sweep schedule id
- current retry-sweep due timestamp
- current DLQ-purge schedule id
- current DLQ-purge due timestamp

#### SQLite / Drizzle
Continue to own:
- pending delivery records
- DLQ records
- attempts / error text / timestamps
- all admin-queryable history-like data

#### Transient in-memory only
Okay to keep non-durable:
- local helper state if needed for one invocation only
- short-lived loop variables / scratch parsing state

### Explicit anti-goal
Do **not** move pending payloads or DLQ payloads into Agent state.

They are query-oriented, admin-facing, and already correctly modeled in SQLite.

## 5. Method-by-method implementation plan

### 5.1 Class/bootstrap conversion

Target shape:
- `class _EventBusDO extends Agent<Env, EventBusAgentState>`
- constructor only initializes Drizzle (`drizzle(this.ctx.storage, { schema })`)
- bootstrap moves to `onStart()`

`initialState` should initialize all schedule tracking fields to `null`.

`onStart()` must run inside `await this.ctx.blockConcurrencyWhile(async () => { ... })` and should:
1. run Drizzle migrations
2. clear legacy raw DO alarm state via `await this.ctx.storage.deleteAlarm()` **before** restoring Agent schedules
3. reconcile any stale schedule ids stored in Agent state
4. inspect `pending_events` for the earliest `nextRetryAt`
5. inspect `dead_letter_queue` for the earliest `expiresAt`
6. restore or cancel retry/purge schedules so Agent state matches the DB

### 5.2 `publish(event)`

Keep current contract:
1. validate with `EventSchema.safeParse`
2. resolve route synchronously from `EVENT_ROUTES`
3. attempt inline delivery immediately
4. if delivery succeeds → return `Result.ok()`
5. if delivery fails → persist for retry and return `Result.ok()` if persistence succeeds

Error behavior:
- invalid event → `EventBusValidationError`
- missing route → `EventBusRoutingError`
- persistence failure → `EventBusDbError`

Idempotency / duplicates:
- Preserve downstream successful-delivery idempotency in `AchievementsDO`; do **not** add a new success ledger table here.
- Before inserting a retry row, check whether the event id already exists in:
  - `pending_events`
  - `dead_letter_queue`
- If already pending or already in DLQ, log and return `Result.ok()` (accepted/no-op) rather than creating duplicate work.
- If persisting a new pending row, use a conflict-safe insert path (`onConflictDoNothing` or equivalent handling) so races do not turn into spurious failures.

Persistence shape for a newly queued retry:
- `id = event.id`
- `event = JSON.stringify(event)`
- `attempts = 0`
- `nextRetryAt = now + 1s`
- `createdAt = now`

After inserting, call a helper that ensures the retry sweep schedule points at the earliest pending row.

### 5.3 Internal callbacks / helpers

Create explicit Agent-native callbacks:
- `retryDueEventsTick()`
  - delayed schedule callback
  - processes all rows where `nextRetryAt <= now`
  - updates/removes rows and then re-schedules the next earliest pending row
- `purgeExpiredDlqTick()`
  - delayed schedule callback
  - deletes rows where `expiresAt <= now`
  - re-schedules to the next earliest remaining expiry

Supporting internal helpers:
- `ensureRetrySweepSchedule()`
- `ensureDlqPurgeSchedule()`
- `clearRetrySweepSchedule()`
- `clearDlqPurgeSchedule()`
- `processPendingEvent(pending)`
- `scheduleRetrySweepAt(when)`
- `scheduleDlqPurgeAt(when)`
- `deliverEvent(event, handlerKey)`

### 5.4 Retry scheduling strategy

**Choose: single earliest-due sweep schedule (Option B in the original prompt).**

Why:
- pending rows already store `nextRetryAt`
- one schedule per event adds unnecessary id bookkeeping
- admin/debug queries care about DB rows, not individual schedule rows
- a single wakeup for the earliest due row mirrors the current alarm semantics cleanly

Rules:
- Always maintain **at most one** retry sweep schedule in Agent state.
- After any pending-row insert/update/delete, recompute the earliest pending row.
- If there are no pending rows:
  - cancel any tracked retry schedule
  - clear `retrySweepScheduleId` + `retrySweepDueAt`
- If the earliest due timestamp changes:
  - cancel the stale schedule (if present)
  - schedule a new delayed callback to `retryDueEventsTick`
  - persist the new id and due timestamp in Agent state
- On startup:
  - if stored id exists and still matches the correct earliest due timestamp, keep it
  - otherwise cancel/recreate from DB truth

Callback retry semantics:
- **Do not rely on Agent built-in callback retries for domain delivery semantics.**
- Schedule `retryDueEventsTick` and `purgeExpiredDlqTick` with `retry: { maxAttempts: 1 }` (or ensure the callback swallows/logs per-row failures) so EventBus retry timing remains governed by the domain model, not hidden 100ms–3000ms Agent callback retries.

### 5.5 Retry attempt behavior

Within `processPendingEvent(pending)`:
1. parse persisted JSON with `EventSchema`
2. if corrupted/unparseable:
   - log error
   - delete the pending row
   - do not send to DLQ (matches current “corrupted pending event, delete it” behavior)
3. compute `attemptNumber = pending.attempts + 1`
4. attempt delivery
5. on success:
   - delete pending row
6. on failure and `attemptNumber < MAX_ATTEMPTS`:
   - update `attempts = attemptNumber`
   - update `nextRetryAt = now + backoff(attemptNumber)`
7. on failure and `attemptNumber >= MAX_ATTEMPTS`:
   - atomically insert into DLQ and delete from pending
   - `attempts = attemptNumber`
   - `firstFailedAt = pending.createdAt`
   - `lastFailedAt = now`
   - `expiresAt = now + 30 days`
   - then ensure DLQ purge schedule is updated

### 5.6 DLQ purge behavior

Use **one dedicated delayed schedule to the earliest `expiresAt`**.

Why this is preferred over daily cron or piggybacking:
- no unnecessary periodic wakeups when the DLQ is empty
- expiry timing stays exact
- purge still happens even when there are no pending retries and no new publishes
- keeps all cleanup scheduling explicit and deterministic

Rules:
- whenever a DLQ row is created or replay/delete changes DLQ contents, recompute the earliest expiry
- maintain at most one DLQ purge schedule
- if no DLQ rows remain, cancel the purge schedule and clear state

### 5.7 Admin/debug RPC methods

Keep these public methods unchanged:
- `getPendingCount()`
- `getPending()`
- `getDLQ()`
- `deleteDLQ()`

`replayDLQ(id)` decision:
- **Preserve current behavior:** attempt inline delivery immediately
- if success: delete from DLQ and return `{ success: true, eventId }`
- if failure: update `lastFailedAt` and `error`, leave the row in DLQ, and return `{ success: false, eventId, error }`

Why keep replay inline instead of moving back to pending:
- admin replay is an explicit operator action meaning “try this now”
- the item has already exhausted automatic retries once
- preserving current semantics avoids silently re-entering an automatic retry cycle that operators did not ask for
- matches current admin route response wording and expectations

## 6. Schema and data migration assessment

### Recommendation: keep schema unchanged

The existing tables should survive unchanged:
- `pending_events`
- `dead_letter_queue`

### Why no new columns are needed
- `handler_key`
  - unnecessary; route is still derivable from `event.type` + `EVENT_ROUTES`
- `scheduled_task_id`
  - unnecessary in DB because the design intentionally uses **one sweep schedule**, with ids kept in Agent state only
- `delivery_status`
  - already represented structurally by pending-vs-DLQ table location
- `last_error_tag` / `last_error_json`
  - nice-to-have for observability, but not worth migration churn for this task

### Migration note
Even though the DB schema stays the same, the implementation should still do a one-time **legacy alarm cleanup** in `onStart()` so old raw DO alarms do not survive the migration.

## 7. Testing strategy

### Harness constraints to respect
- Call `await stub.setName("event-bus")` in Agent tests before exercising state/schedules.
- DO-to-DO RPC from inside `runInDurableObject()` is unreliable in this repo’s worker-pool harness.
- Prefer public behavior and durable state inspection.
- Avoid production test seams / `vi.mock()`.

### Preferred local test strategy
Use the real EventBus public methods plus durable-state inspection. For schedule-specific cases, seed or inspect durable rows inside `runInDurableObject()` rather than adding test-only production APIs.

Because EventBus → Achievements DO success is not faithfully testable in miniflare, local automated tests should lean on:
- validation/admin/public RPC behavior
- durable pending/DLQ state inspection
- schedule restoration / cancellation behavior
- callback execution against seeded rows

### Targeted automated tests

#### Contract tests
- invalid event payload returns `EventBusValidationError`
- malformed UUID returns `EventBusValidationError`
- valid event whose handler delivery fails in the harness still makes `publish()` return `ok` when the event is accepted for retry persistence
- route completeness is protected without production seams:
  - either via a small extracted pure route resolver test
  - or a compile-time/runtime completeness assertion around `EVENT_ROUTES satisfies Record<EventType, ...>`

#### Retry orchestration tests
- publishing a valid event creates one pending row and a retry sweep schedule when inline delivery fails
- `retryDueEventsTick()` increments attempts and pushes `nextRetryAt` forward for a due row below max attempts
- a due row already at `attempts = 2` moves to DLQ on another failed attempt
- corrupted pending JSON is deleted during retry processing
- `purgeExpiredDlqTick()` removes expired DLQ rows
- `replayDLQ()` success/failure behavior matches the chosen contract (in local harness, failure path is the practical automated case)
- `deleteDLQ()` deletes existing rows and returns not-found for missing ids

#### Agent-specific tests
- `onStart()` restores retry sweep schedule from existing `pending_events`
- `onStart()` restores DLQ purge schedule from existing `dead_letter_queue`
- stale schedule ids in state are reconciled
- no duplicate retry sweep schedules are created when multiple events are queued
- retry sweep schedule is cancelled/cleared when pending becomes empty
- DLQ purge schedule is cancelled/cleared when DLQ becomes empty

### How to seed due/expired rows in tests
Prefer one of these, in order:
1. seed durable rows inside `runInDurableObject()` using the Agent/SQLite surface already available there
2. assert via public RPC methods (`getPending`, `getDLQ`, `getSchedules`)
3. avoid adding any production-only helper methods just for tests

### Success-path integration test recommendation
Because real EventBus → Achievements DO RPC success is the exact thing miniflare is bad at here, prefer:
- **one focused deployed/manual integration smoke test against real workerd**

Suggested manual smoke:
1. publish a valid event to EventBus in a dev/staging deployment
2. verify no pending row is created on success
3. verify Achievements/event_history reflects the event once

This is preferred over adding a fake handler seam to production code.

## Notes for the implementer
- The key migration insight is: **Agent adds better lifecycle and delayed-task primitives, but SQLite remains the EventBus source of truth.**
- Do not force queue/workflow usage just because the SDK provides them.
- The smallest coherent, idiomatic Agent migration here is: **Agent state for schedule coordination + SQLite for durable bus records + one-shot delayed schedules for next-due work.**
