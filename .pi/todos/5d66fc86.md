Investigate production alarm exception in SongQueueDO on deployed version 7c11a66a-9cae-4be1-8e90-4ce7450d670f. Determine root cause from code/tests/logs and propose or implement fix.

Confirmed via Workers Observability that SongQueueDO alarm exceptions are still a PartyServer/Agent name bootstrap failure on cold alarm wakeups. Queried telemetry for `Attempting to read .name on _SongQueueDO...` and recovered stack traces showing the throw occurs in PartyServer's `get name()` during Agent `onStart()` bootstrap (`this.mcp.restoreConnectionsFromStorage(this.name)`), before schedule callback execution. Implemented a stronger `getStub()` workaround in `apps/api/src/lib/durable-objects.ts`: initialize Agent-backed stubs through PartyServer's `fetch`-based set-name route (`/cdn-cgi/partyserver/set-name/`) before falling back to RPC `setName()`. This targets cold-start/alarm persistence more directly than RPC-only naming. Validated with targeted tests and lint/format.

Found a second SongQueueDO scheduling bug while debugging alarm exceptions. `refreshQueueTick()` / `cleanupStalePendingTick()` only nulled tracked schedule state; `clearRefreshSchedule()` / `clearCleanupSchedule()` canceled only the tracked schedule id, not orphaned same-callback rows. Repeated rescheduling could leave multiple `refreshQueueTick` rows alive, producing the observed fractional alarm cadence and repeated extra alarms. Patched SongQueueDO to cancel all same-callback schedules before rescheduling/clearing, and added regression coverage in `song-queue-do.test.ts` to assert only one refresh schedule remains after repeated `refreshQueueTick()` rescheduling. Validated with targeted vitest, typecheck, oxlint, and oxfmt.
