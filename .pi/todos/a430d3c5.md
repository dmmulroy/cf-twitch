## Goal

Migrate `apps/api/src/durable-objects/spotify-token-do.ts` from `DurableObject<Env>` to `Agent<Env, SpotifyTokenAgentState>`.

Use these as the primary playbooks/templates:

- `docs/durable-object-agent-migration-guide.md`
- `apps/api/src/durable-objects/twitch-token-do.ts`
- `apps/api/src/__tests__/durable-objects/twitch-token-do.test.ts`

This should be treated as a bounded follow-up to the completed Twitch migration, not a generalized token abstraction project.

---

## Why this should be straightforward now

`SpotifyTokenDO` is the near-clone follow-up to `TwitchTokenDO`.

The Twitch migration already established the desired pattern for token-like DOs in this repo:

- bootstrap in `onStart()`
- Agent state as the single source of truth for current token state
- one-time migration from legacy `token_set`
- one-shot Agent scheduling instead of alarms
- public scheduled callback for refresh behavior
- tests kept at the public-behavior level

So the Spotify migration should mostly be a careful adaptation of the Twitch Agent pattern to Spotify-specific refresh semantics.

## Current `SpotifyTokenDO` design summary

### File
- `apps/api/src/durable-objects/spotify-token-do.ts`

### Current class shape
- `class _SpotifyTokenDO extends DurableObject<Env> implements StreamLifecycleHandler<TokenError>`

### Current public RPC methods to preserve
1. `onStreamOnline(): Promise<Result<void, TokenError>>`
2. `onStreamOffline(): Promise<Result<void, TokenError>>`
3. `getValidToken(): Promise<Result<string, TokenError>>`
4. `setTokens(tokens: SpotifyTokenResponse): Promise<Result<void, never>>`

### Current scheduled/lifecycle entrypoint
- `alarm(): Promise<void>`

### Current state model

#### In-memory / process-local today
- `tokenCache: TokenSet | null`
- `refreshPromise: Promise<Result<string, TokenError>> | null`

#### SQLite / Drizzle today
The singleton `token_set` row stores:
- `accessToken`
- `refreshToken`
- `tokenType`
- `expiresIn`
- `expiresAt`
- `isStreamLive`

#### Additional raw DO storage key today
- `alarmRetryCount`

### Current runtime behavior

#### Constructor bootstrap today
The constructor:
- initializes Drizzle
- runs migrations
- loads the singleton `token_set` row into cache
- proactively refreshes on startup if stream is live and token is expired/in the refresh window
- reschedules an alarm if stream is live

#### Online transition today
`onStreamOnline()`:
- marks `isStreamLive = true` in SQLite
- mirrors it in cache
- refreshes immediately if needed
- schedules proactive refresh alarm

#### Offline transition today
`onStreamOffline()`:
- marks `isStreamLive = false`
- mirrors it in cache
- cancels alarm
- clears `alarmRetryCount`

#### Token reads today
`getValidToken()`:
- returns cached token if valid
- errors if no token
- errors if expired and stream offline
- refreshes if expired and stream live
- coalesces concurrent refreshes with `refreshPromise`

#### Token writes today
`setTokens()`:
- computes `expiresAt`
- persists to `token_set`
- preserves previous `refresh_token` if provider response omits it
- updates cache
- schedules refresh if stream is live

#### Alarm behavior today
`alarm()`:
- reloads token from DB
- exits if stream offline
- refreshes once
- clears retry state on success
- uses exponential backoff retry for retryable network failures:
  - 60s
  - 120s
  - 240s
- falls back to 10 minutes after exhaustion/non-retryable failure

## Desired migration design

## 1. Use Agent state as the only long-term source of truth for current state

This DO is a current-state DO, not a history/query DO.

Use Agent state for:
- current token
- `isStreamLive`
- `refreshScheduleId`
- `refreshRetryCount`

Keep only `refreshPromise` as transient in-memory state.

Do not keep long-term dual-write or dual-read behavior with `token_set`.

### Suggested state shape

```ts
interface SpotifyTokenState {
  accessToken: string;
  refreshToken: string;
  tokenType: string;
  expiresIn: number;
  expiresAt: string;
}

interface SpotifyTokenAgentState {
  token: SpotifyTokenState | null;
  isStreamLive: boolean;
  refreshScheduleId: string | null;
  refreshRetryCount: number;
}
```

Exact naming can vary, but keep the same ownership model as `TwitchTokenDO`.

---

## 2. Move bootstrap to `onStart()`

Follow the Twitch pattern:

```ts
async onStart(): Promise<void> {
  await this.ctx.blockConcurrencyWhile(async () => {
    await migrate(this.legacyDb, migrations);
    await this.migrateLegacyStateOnce();
    await this.clearLegacyAlarmState();
    await this.restoreOrRecomputeRefreshSchedule();
  });
}
```

Responsibilities:
- run legacy token schema migrations
- migrate legacy `token_set` row once into Agent state if needed
- delete legacy row after hydration
- clear old alarm state (`deleteAlarm()` and `alarmRetryCount`)
- restore/cancel the correct Agent schedule
- if a live token is already in the refresh window, refresh immediately or schedule an immediate refresh

Do not keep constructor-time bootstrap as the real initialization path.

---

## 3. Replace alarms with one-shot Agent scheduling

Do not use `scheduleEvery()` here.

Spotify token refresh timing is driven by `expiresAt`, so use one-shot `schedule()` plus `cancelSchedule()`.

Recommended pattern:
- store `refreshScheduleId` in Agent state
- create a public scheduled callback such as `refreshTokenTick()`
- on token updates / stream live changes:
  - cancel old schedule if needed
  - compute exact next refresh time
  - create a new schedule
  - persist its id in Agent state

Preserve behavior:
- refresh buffer: 5 minutes before expiry
- retry base delay: 60 seconds
- max retries: 3
- fallback delay: 10 minutes

---

## 4. Preserve public RPC contract

Keep these methods and semantics:
- `onStreamOnline()`
- `onStreamOffline()`
- `getValidToken()`
- `setTokens()`

Continue using:
- `@rpc`
- `Result<T, E>`
- existing token error types

Do not add browser/client callable APIs.
Do not add `onRequest()` unless some real HTTP compatibility surface appears.

---

## 5. Preserve Spotify-specific refresh behavior

Unlike Twitch, Spotify refresh requests use:
- `POST https://accounts.spotify.com/api/token`
- form body with `grant_type=refresh_token`
- `Authorization: Basic ...` header using client id/secret

Preserve current refresh-token retention behavior when Spotify omits `refresh_token` from refresh responses.

That behavior is important and already covered by tests.

---

## 6. Testing expectations

Adapt the Spotify tests to match the Twitch Agent pattern.

### Existing tests to preserve/update
- initial no-token behavior
- cached token reads while offline
- online refresh of expired token
- refresh-token retention when response omits `refresh_token`
- concurrent refresh coalescing
- refresh error handling
- persistence across instance restarts

### Add/adjust tests for Agent scheduling
- public scheduled callback success path
- retryable failure schedules 60s retry
- non-retryable parse/schema failure schedules 10m fallback
- offline transition cancels scheduled refresh work

### Important harness lessons from the Twitch migration
- set the Agent stub name in test setup:
  - `await stub.setName("spotify-token")`
- validate scheduled behavior via public/scheduled methods
- do not over-assert exact schedule counts/order when manually invoking scheduled callbacks; prefer `arrayContaining/objectContaining` unless the test is specifically about cancellation

## Acceptance criteria

1. `SpotifyTokenDO` extends `Agent<Env, SpotifyTokenAgentState>`
2. bootstrap/migration logic lives in `onStart()` with `ctx.blockConcurrencyWhile(...)`
3. Agent state is the single source of truth for current Spotify token state
4. `alarm()` is removed
5. raw `ctx.storage.setAlarm()` / `deleteAlarm()` usage is removed from this DO
6. proactive refresh uses Agent scheduling APIs
7. public RPC methods and Result/error behavior remain compatible
8. refresh-token retention still works when Spotify omits `refresh_token`
9. targeted Spotify token tests pass
10. typecheck passes
11. legacy compatibility is one-time only, not permanent dual-read/dual-write

---

## Key warning

Do not turn this into a shared token abstraction unless the shared helper is trivially obvious after the migration is complete.

The primary goal is:
- migrate `SpotifyTokenDO` cleanly using the proven Twitch pattern
- preserve public behavior
- avoid long-lived compatibility baggage
- keep the change easy to review
