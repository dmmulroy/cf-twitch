Investigate and correctly fix failing integration tests that are currently unrelated to the KeyboardRaffleDO migration.

## Observed failures

### 1) `apps/api/src/__tests__/durable-objects/keyboard-raffle-saga-do.test.ts`
Command:
- `cd apps/api && pnpm test src/__tests__/durable-objects/keyboard-raffle-saga-do.test.ts`

Observed:
- test `resumes a pending retry via the scheduled callback and completes the saga`
- and `restores a pending retry schedule from durable saga rows during startup`
- fail in global teardown with pending `fetchMock` interceptors for `POST https://api.twitch.tv/helix/chat/messages`

Strong diagnosis:
- The test sets up too many chat mocks:
  - one `mockTwitchRedemptionUpdate(fetchMock)`
  - four `mockTwitchChatMessage(fetchMock)`
- In the exercised path, only one chat message appears to actually be sent.
- Because `apps/api/src/__tests__/setup.ts` calls `fetchMock.assertNoPendingInterceptors()` after each test, the unused chat mocks fail the test.

### 2) `AchievementsDO` / `EventBusDO` transitive Agent startup problem
Commands:
- `cd apps/api && pnpm test src/__tests__/durable-objects/event-bus-do.test.ts`
- `cd apps/api && pnpm test src/__tests__/durable-objects/keyboard-raffle-saga-do.test.ts`

Observed logs:
- `AchievementsDO: Handling event`
- then error from `recordToEventHistory`
- `no such table: event_history: SQLITE_ERROR`

Important context:
- `apps/api/src/__tests__/durable-objects/achievements-do.test.ts` explicitly names and warms the Agent singleton before use:
  - `await stub.setName(name)`
  - `await stub.getDefinitions()`
- In the failing transitive paths, `AchievementsDO` is invoked indirectly through:
  - `KeyboardRaffleSagaDO -> EventBusDO -> getStub("ACHIEVEMENTS_DO") -> handleEvent(...)`
- `EventBusDO` tests similarly do not appear to explicitly initialize `AchievementsDO` before expecting successful delivery.

High-confidence hypothesis:
- In worker-pool tests, transitive Agent singleton startup/migration is not reliably established unless the test explicitly names/warms the Agent first.
- `AchievementsDO.onStart()` migrations are therefore not guaranteed to have run before `handleEvent()` reaches `event_history`.

## Suggested fix direction

Do not apply a quick hack. Fix the tests by making Agent initialization explicit and consistent.

Likely steps:
1. Add a small test helper to initialize `AchievementsDO` in tests that expect `EventBusDO` delivery success or saga-driven event publication.
   - likely pattern:
     - `const id = env.ACHIEVEMENTS_DO.idFromName("achievements")`
     - `const stub = env.ACHIEVEMENTS_DO.get(id)`
     - `await stub.setName("achievements")`
     - `await stub.getDefinitions()`
2. Update `event-bus-do.test.ts` setup for cases that expect immediate successful delivery or successful DLQ replay.
3. Update `keyboard-raffle-saga-do.test.ts` setup to initialize `AchievementsDO` if the scenario expects the event path to work during the test.
4. Reduce `mockTwitchChatMessage(fetchMock)` calls in saga tests to exactly match expected sends for the exercised path.
5. Re-run at least:
   - `cd apps/api && pnpm test src/__tests__/durable-objects/event-bus-do.test.ts`
   - `cd apps/api && pnpm test src/__tests__/durable-objects/keyboard-raffle-saga-do.test.ts`
   - `cd apps/api && pnpm test src/__tests__/durable-objects/achievements-do.test.ts`
