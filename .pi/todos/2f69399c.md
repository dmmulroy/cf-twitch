## Goal

Migrate `apps/api/src/durable-objects/song-queue-do.ts` from `DurableObject<Env>` to `Agent<Env, SongQueueAgentState>` following `docs/durable-object-agent-migration-guide.md`, while preserving the current RPC contract and the existing lazy Spotify sync behavior.

## Design summary

`SongQueueDO` is **not** a token-style DO where all live data should move into Agent state.
The best fit is closer to the `EventBusDO` pattern:

- **SQLite / Drizzle remains the durable source of truth** for queue data and reconciliation state:
  - `pending_requests`
  - `spotify_queue_snapshot`
  - `request_history`
- **Agent state only coordinates current freshness metadata**, not the queue contents:
  - `lastSyncAt: string | null`
- **Transient in-memory fields stay process-local**:
  - `syncLock: Promise<Result<void, SongQueueDbError>> | null`

Rationale:

- reconciliation logic depends on durable queue snapshots across restarts (`previous position 0`, seen timestamps, dropped-track cleanup)
- analytics/history queries already belong in SQLite
- the current Spotify snapshot is better treated as a durable cached working set than copied into Agent state
- the existing DO has **no alarm-driven behavior**, so this migration should preserve the current read-through `ensureFresh()` model rather than invent new schedules

## Public contract to preserve

Keep these RPC methods unless an explicit cleanup review says otherwise:

- `persistRequest(request)`
- `deleteRequest(eventId)`
- `deleteHistory(eventId)`
- `writeHistory(eventId, fulfilledAt)`
- `getCurrentlyPlaying()`
- `getQueue(limit?)`
- `getRequestHistory(limit?, offset?, since?, until?)`
- `getSessionRequestCount(since)`
- `getUserRequestCount(userId)`
- `getUserRequestCountByDisplayName(displayName)`
- `getTopTracks(limit?)`

- `checkDuplicateRequest(userId, trackId, windowMinutes?)`

Cleanup candidate:

- `ping()` appears test/debug-only. Verify there are no real callers. If none, remove it during migration instead of preserving legacy debug surface.

Keep all existing tables and migrations intact:

### Transient

## Implementation plan

### 1. Refactor class shape to Agent

- Change base class to `Agent<Env, SongQueueAgentState>`
- Update constructor signature to `(ctx: AgentContext, env: Env)`
- Keep `this.db = drizzle(this.ctx.storage, { schema })`
- Retain `withRpcSerialization(...)` export pattern
- Add `initialState = { lastSyncAt: null }`

### 2. Move bootstrap into `onStart()`

Implement:

```ts
async onStart(): Promise<void> {
  await this.ctx.blockConcurrencyWhile(async () => {
    await migrate(this.db, migrations);
    await this.hydrateLastSyncAtFromSnapshot();
  });
}
```

Notes:

- there is no legacy `ctx.storage.put()` row to migrate
- startup work is mostly DB migration plus one-time hydration of Agent freshness state
- `hydrateLastSyncAtFromSnapshot()` should read an existing snapshot row (all rows share the same `syncedAt`) and restore `state.lastSyncAt`
- if no snapshot exists, leave `lastSyncAt = null`

### 3. Keep lazy sync behavior; do **not** add schedules in v1

Preserve the current model:

- reads call `ensureFresh()`
- freshness window stays `15_000ms` unless a separate product/perf reason changes it
- no `schedule()` / `scheduleEvery()` should be introduced just because the class is now an Agent

This migration is about lifecycle/state ownership, not changing the queue-refresh strategy.

### 4. Convert freshness coordination to Agent state

Refactor `ensureFresh()` to use `this.state.lastSyncAt` instead of the old in-memory `number | null`.

Behavior to preserve:

- if snapshot is fresh, return immediately
- if a sync is already in flight, await `syncLock`
- on successful sync, write `lastSyncAt = nowIso`
- on sync failure, keep stale DB snapshot and return `ok()`

Implementation detail:

- compare timestamps using `new Date(this.state.lastSyncAt).getTime()`
- keep `syncLock` transient; do **not** persist an in-flight flag in Agent state

### 5. Update invalidation points

Any mutation that previously did `this.lastSyncAt = null` should now do:

```ts
this.setState({ ...this.state, lastSyncAt: null });
```

At minimum:

- `persistRequest()` invalidates freshness so the next queue read forces a new attribution pass

Also review whether any other write paths should invalidate cached freshness for consistency.

### 6. Keep SQLite as source of truth for reconciliation

Do **not** move these into Agent state:

- current Spotify snapshot rows
- pending request pools
- request history
- seen timestamps / reconciliation bookkeeping

Keep existing reconciliation flow intact:

- read previous position 0 from `spotify_queue_snapshot`
- match pending requests FIFO per `trackId`
- reconcile played track
- replace snapshot rows
- update seen timestamps
- reconcile dropped tracks
- cleanup stale pending

This is the core behavior to preserve.

### 7. Cleanup pass during migration

- Audit and likely remove `ping()` if it is only test scaffolding
- Remove any DurableObject-era comments that imply constructor-time bootstrap is still the source of truth
- Update top-of-file docs to describe the Agent/SQLite split clearly
- Ensure there is no fetch/onRequest compatibility surface added unnecessarily

## Testing plan

Follow the migration guide:

- prefer **public stub calls** for observable behavior
- use `runInDurableObject(...)` only for direct DB seeding or internal Agent state inspection when necessary
- avoid adding test-only production seams

1. **CRUD/history/statistics tests**
   - migrate existing tests to work against Agent stubs
   - use public stub calls where possible
   - keep direct DB assertions only where needed

2. **Startup hydration test**
   - seed `spotify_queue_snapshot` directly through Drizzle
   - start/reopen named Agent stub
   - verify `lastSyncAt` is restored from the existing snapshot so a fresh cached snapshot can be reused

3. **Freshness invalidation test**
   - seed a fresh snapshot
   - call `persistRequest()`
   - verify the next queue read attempts a sync instead of trusting the old freshness state

4. **Stale fallback test**
   - seed a stale snapshot
   - mock Spotify failure
   - verify `getCurrentlyPlaying()` / `getQueue()` still return from stale snapshot instead of surfacing an error

5. **Sync path tests**
   - replace the currently skipped sync coverage with public-stub-based tests where feasible
   - set up `SpotifyTokenDO` via its public API and `fetchMock`
   - exercise `getCurrentlyPlaying()` and `getQueue()` through the `SongQueueDO` stub, not from inside nested `runInDurableObject(...)`

6. **Reconciliation behavior tests**
   - pending request becomes history when prior position 0 disappears
   - previously-seen unmatched requests are dropped
   - never-seen pending requests are not deleted prematurely

### Regression focus

Be extra careful around:

- FIFO matching when multiple pending requests share the same `trackId`
- preserving stale fallback semantics on Spotify/API errors
- ensuring history and pending mutations still remain atomic inside a single RPC call

## Risks / open questions

1. **What should count as the durable freshness source of truth?**
   Preferred answer: Agent state holds the authoritative freshness timestamp, but startup hydrates it from `spotify_queue_snapshot.syncedAt` to preserve behavior across restarts.

2. **Should `ping()` survive?**
   Preferred answer: remove it unless a real caller exists.

3. **Can we fully unskip Spotify sync tests after the Agent migration?**
   Probably yes for public-stub paths, but verify in worker-pool tests before depending on that assumption.

## Out of scope for this migration

- redesigning the Spotify attribution algorithm
- moving queue snapshot contents into Agent state
- introducing schedules for background sync/cleanup
- changing saga contracts or webhook command behavior
- optimizing query performance unless the migration uncovers a concrete issue

## Agent-first revision (per follow-up)

This migration should be **more agent-native** than a minimal base-class swap. The design/implementation should explicitly evaluate and, where it genuinely helps, adopt the Agent runtime features instead of only preserving the old lazy-read behavior.

### Guiding principle

“Take full advantage of Agent” does **not** mean force every API into the design.
It means:

- use Agent state for fast-changing coordination state
- use `onStart()` as the single bootstrap point
- use Agent scheduling where it improves freshness/cleanup behavior
- evaluate workflows/queues/web handlers deliberately and document why they are or are not the right fit

## Revised Agent capability plan

### 1. Use Agent state for **operational coordination**, not just `lastSyncAt`

Expand the design review to consider a small but useful coordination state such as:

```ts
interface SongQueueAgentState {
	lastSyncAt: string | null;
	refreshScheduleId: string | null;
	refreshDueAt: string | null;
	cleanupScheduleId: string | null;
	cleanupDueAt: string | null;
	consecutiveSyncFailures: number;
}
```

Expected direction:

- keep queue contents/history in SQLite
- keep Agent state focused on freshness and scheduling coordination
- persist only the small coordination fields that benefit from Agent durability across restarts

### 2. Add **Agent-native scheduling** where it materially improves the queue

Instead of preserving a purely read-triggered sync model by default, explicitly evaluate these schedules:

#### a. one-shot refresh schedule after queue mutations

After `persistRequest()` (and possibly other invalidating writes), schedule a near-term refresh tick so the cache/snapshot can be updated proactively instead of waiting for the next reader.

Use case:

- the request saga just queued a song
- chat/overlay/API reads immediately after should see updated attribution with less read-path latency

Preferred shape:

- `schedule(delaySeconds, "refreshQueueTick")` or a specific `Date`
- store/cancel the schedule id in Agent state to avoid duplicate pending refreshes

#### b. self-sustaining refresh while queue activity exists

Evaluate keeping a periodic refresh schedule active only while it is useful, e.g. while:

- there are pending requests, or
- Spotify currently has tracks in snapshot, or
- the currently playing track is a user-requested song that may need reconciliation soon

This would let the Agent keep queue state warm and reconcile played/dropped songs without relying entirely on external reads.

Expected design constraint:

- **do not** blindly poll forever
- schedule only while there is active music/request activity
- cancel schedules when the system becomes idle

#### c. scheduled stale-pending cleanup

`cleanupStalePending()` is a natural candidate for an Agent scheduled callback instead of only piggybacking on syncs.

Evaluate a low-frequency cleanup tick (for example hourly or every few minutes while active) so stale pending rows are cleaned even during quiet read periods.

### 3. Keep `ensureFresh()` as a read-path safety net, not the only sync trigger

Even with proactive schedules, reads should still call `ensureFresh()` so:

- cold starts still self-heal
- missed schedules do not break correctness
- stale snapshot fallback continues to work

Target shape:

- schedules improve freshness/latency
- `ensureFresh()` preserves correctness

### 4. Evaluate Agent workflows/queues explicitly, then likely reject for this DO

Add an explicit design decision section in the final implementation notes:

- **Agent workflows**: likely unnecessary because SongQueueDO is already a singleton coordinator and its sync/reconciliation logic is short-lived and idempotent enough to stay as plain methods/scheduled callbacks.
- **Agent queues**: likely not a fit because this is not a fan-out/FIFO work-consumer problem; ordering is driven by current Spotify state + DB reconciliation, not by an append-only task queue.
- **onRequest()**: only add if there is a genuine compatibility or debugging HTTP surface that should remain public; otherwise keep RPC-only.

The important requirement is to **evaluate** these features and document why the chosen design uses scheduling/state heavily but not queues/workflows if they do not improve the model.

## Revised implementation phases

### Phase 1 — Agent conversion + bootstrap

- convert `SongQueueDO` to `Agent`
- move migrations/bootstrap into `onStart()`
- hydrate coordination state from durable snapshot rows if appropriate
- restore or cancel any refresh/cleanup schedules during startup

### Phase 2 — Agent-native background coordination

- introduce `refreshQueueTick()` scheduled callback
- introduce `cleanupStalePendingTick()` scheduled callback if chosen
- add helpers like `ensureRefreshSchedule()`, `cancelRefreshSchedule()`, `ensureCleanupSchedule()`
- make invalidating writes schedule refresh work instead of only nulling freshness state

### Phase 3 — keep read-path correctness

- preserve `ensureFresh()` with stale fallback semantics
- coalesce concurrent syncs with transient `syncLock`
- ensure scheduled and read-triggered syncs share the same internal sync path

### Phase 4 — explicit design cleanup

- remove `ping()` if unused
- remove any unnecessary compatibility/debug leftovers
- add comments/docstrings that clarify the Agent-vs-SQLite split and why queues/workflows were not adopted if rejected

## Additional test expectations for Agent-native features

Add/adjust tests for:

- startup restoring refresh/cleanup schedules from Agent state / durable DB conditions
- invalidating writes scheduling a refresh tick
- scheduled refresh keeping snapshot warm without a reader
- idle-state schedule cancellation
- scheduled cleanup deleting stale pending rows independently of reads
- `ensureFresh()` still working correctly when schedules are absent or delayed

## Decision standard

Use the Agent feature set aggressively **where it improves freshness, latency, and autonomous reconciliation**.
Do **not** use Agent features ceremonially.
The migration should read like an Agent-native redesign of coordination behavior, while still keeping SQLite as the durable source of truth for queue/history data.
