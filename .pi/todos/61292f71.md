## Goal

Produce a **design-first, implementation-ready migration plan** for moving `apps/api/src/durable-objects/event-bus-do.ts` from `DurableObject<Env>` to `Agent<Env, State>`.

This is **not** a 1:1 port task. The main objective is to determine how to use the Agents SDK idiomatically so we delete bespoke infrastructure where possible, especially around:

- alarm orchestration
- retry scheduling
- delayed delivery bookkeeping
- background execution
- future extensibility toward richer routing/orchestration

Do **not** start implementation as part of this todo. The deliverable is the design package / migration todo itself.

## Phase 2 — Decide the Agent-native architecture, not just the class conversion

Produce an explicit design comparison with pros/cons for **at least three candidate architectures**.

### Candidate A — Minimal Agent port (state + schedule, no queue)
Shape:
- `EventBusDO` becomes `Agent<Env, EventBusAgentState>`
- preserve current DB tables
- replace `alarm()` with one-shot `schedule()` for next due retry/purge sweep
- keep retry processing loop inside the agent

Pros to evaluate:
- lowest migration risk
- closest behavioral parity
- easy to preserve admin/debug APIs

Cons to evaluate:
- may preserve too much bespoke retry orchestration
- still manually scans pending rows and computes next retry schedule

### Candidate B — Agent queue + schedule hybrid (**likely preferred default to evaluate hardest**)
Shape:
- keep SQLite as durable source of truth for pending + DLQ + audit-style metadata
- use `queue()` for immediate asynchronous delivery work or controlled reprocessing units
- use `schedule()` only for delayed retries / DLQ purge wakeups
- use Agent state only for small runtime coordination state such as active schedule ids / sweep status

Questions to answer:
- Should `publish()` attempt direct delivery inline, or immediately `queue()` a delivery task?
- Should retries be represented as DB rows only, with scheduled sweeps enqueueing ready deliveries?
- Should each failed event get its own delayed schedule, or should there be one “retryDueEventsTick” schedule for the earliest due row?
- Does queue FIFO help or hurt EventBus semantics given that due time, not insertion time, is the primary ordering constraint?

### Candidate C — Workflow-backed delivery orchestration
Shape:
- one workflow instance per event delivery/retry lifecycle
- workflow handles attempts/backoff/exhaustion
- agent retains admin APIs, routing config, and possibly DLQ indexing/querying

Questions to answer:
- Is a workflow overkill for a single-handler, 3-attempt exponential retry loop?
- Would workflows materially simplify logic, or just move simple retry state into a heavier runtime?
- Are workflows better reserved for sagas rather than the bus itself?

### Required architectural decision
End Phase 2 with a written recommendation that explicitly chooses one of:
- schedule-only Agent design
- queue + schedule hybrid
- workflow-backed design

The recommendation must explain **why** it best matches this repo’s EventBus responsibilities.

## Phase 3 — Define the target state model and persistence split

Assuming the chosen design uses `Agent`, define a precise ownership split.

### Agent state should stay small and operational
Propose a concrete state type, for example:

```ts
interface EventBusAgentState {
  retrySweepScheduleId: string | null;
  dlqPurgeScheduleId: string | null;
  isRetrySweepQueued: boolean;
  isPurgeSweepQueued: boolean;
}
```

This is only an example — refine it based on the chosen architecture.

### Rules for state ownership
Document which data belongs where:

#### Agent state
Only small current operational coordination data, such as:
- active schedule ids
- maybe a “sweep already queued” flag
- maybe migration version / bootstrap guard only if necessary

#### SQLite / Drizzle
Retain durable/queryable records in DB:
- pending delivery records
- DLQ records
- timestamps / attempts / last error text
- any future admin query surface

#### Transient in-memory only
Keep process-local helpers out of persistent state, for example:
- in-flight delivery promise map if one is added
- short-lived reentrancy guard if safe to lose on hibernation

### Explicit anti-goal
Do **not** move all pending/DLQ payloads into Agent state. They are query-oriented and admin-facing; they belong in SQLite.

## Phase 4 — Design the actual method-by-method migration

Write the planned target behavior for each public and internal entrypoint.

### 1. Class/bootstrap conversion
Target shape:
- `class _EventBusDO extends Agent<Env, EventBusAgentState>`
- Drizzle stays on `this.ctx.storage`
- bootstrap moves to `onStart()`
- `ctx.blockConcurrencyWhile(...)` wraps migration/bootstrap

`onStart()` responsibilities should be specified exactly:
- run Drizzle migrations
- perform any one-time legacy cleanup/migration if needed
- inspect pending rows and restore the correct schedule(s)
- ensure stale schedule ids in Agent state are reconciled
- optionally schedule a periodic or next-due purge/retry tick depending on final design

### 2. `publish(event)`
Define whether the final design should:
- validate and route synchronously
- attempt inline delivery first, then fall back to queue/schedule
- or enqueue all delivery attempts immediately and return

Whichever path is chosen, specify:
- error behavior for invalid event / missing route
- whether `publish()` remains success-returning when the event is persisted for later retry
- how idempotency is preserved for duplicate event IDs
- where the serialized event payload is persisted

### 3. Delivery callback(s)
Define the internal callback API that Agent scheduling/queueing will invoke.
Examples to consider:
- `deliverEventTask(payload, queueItem?)`
- `retryDueEventsTick()`
- `purgeExpiredDlqTick()`
- `attemptDelivery(eventId)`

Describe exactly what each callback reads/writes and whether it is:
- queued work (`queue()`)
- delayed work (`schedule()`)
- recurring work (`scheduleEvery()`)

### 4. Retry scheduling
Specify one clear strategy:

#### Option A: one schedule per event
- each failed event gets its own delayed schedule
- callback attempts delivery for that event only

#### Option B: single earliest-due sweep schedule
- persist next retry timestamps in DB
- maintain exactly one schedule for the earliest due retry
- callback processes all due rows and schedules the next earliest row

Whichever strategy is chosen, explain:
- duplicate schedule avoidance
- startup restoration behavior
- cancellation/update behavior when rows are deleted or replayed
- how retry storm risk is controlled

### 5. DLQ purge behavior
Decide whether DLQ cleanup should use:
- one dedicated delayed schedule to next expiry
- a daily cron schedule
- piggybacking on retry sweeps

Document why.

### 6. Admin/debug RPC methods
Plan whether these remain unchanged:
- `getPendingCount()`
- `getPending()`
- `getDLQ()`
- `replayDLQ()`
- `deleteDLQ()`

For `replayDLQ()` specifically, decide whether replay should:
- attempt inline delivery immediately
- move the item back to pending and schedule/queue delivery
- preserve the current “try once now, leave in DLQ on failure” behavior

### Known harness constraints to account for
- Agent tests may require `await stub.setName("event-bus")`
- DO-to-DO RPC from inside `runInDurableObject()` is unreliable in miniflare for this repo
- tests should prefer public behavior and durable state inspection
- avoid `vi.mock()` and test-only production seams unless absolutely necessary

### Required test cases to specify

#### Contract tests
- invalid event payload returns `EventBusValidationError`
- unknown route returns `EventBusRoutingError`
- `publish()` returns `ok` when event is accepted for retry/persistence

#### Retry orchestration tests
Depending on final design, specify tests for:
- initial delivery failure persists retry state correctly
- earliest retry schedule is created/restored correctly
- retry callback removes successful events from pending
- max-attempts path moves event to DLQ
- DLQ purge removes expired rows
- replay path behaves exactly as specified

#### Agent-specific tests
- startup bootstrap in `onStart()` restores schedules from existing rows
- duplicate schedule creation is avoided
- schedule cancellation/update logic works when pending becomes empty
- if queue is used: queue callbacks process FIFO units as expected

#### Integration tests to consider later
Because DO-to-DO RPC is hard in worker-pool tests, propose whether one of these is needed:
- a focused deployed/manual integration test against real workerd
- a fake/test handler path strictly for tests (only if reviewable and justified)
- a narrow internal seam around `deliverEvent()` that does not distort the production design

The test plan should explicitly state which approach is preferred.

## Current recommendation bias to validate, not assume

Based on prior discussion, the most likely good target is:
- keep pending/DLQ in SQLite
- use `Agent` for bootstrap + small operational state
- use `schedule()` for delayed wakeups / next-due retry coordination
- evaluate `queue()` carefully for immediate background delivery units
- reserve `workflows` for saga DO replacement rather than forcing them into EventBus

But this is only a starting hypothesis. The todo is not complete until that recommendation is either validated or replaced by a better Agent-native design.
