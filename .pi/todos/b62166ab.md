# Goal

Implement **end-to-end structured logging** across the entire `cf-twitch` application so an engineer can trace a single request/event/redemption through:

- HTTP ingress (`Hono` routes)
- Durable Object RPC boundaries
- Durable Object business logic
- saga orchestration and retries
- outbound Spotify/Twitch API calls
- cache behavior
- analytics write failures
- tail worker exception processing
- optional browser/overlay client failures

This work must be implemented so logs are:

1. **Structured JSON**
2. **Correlatable end-to-end**
3. **Safe** (sensitive fields redacted / not logged)
4. **Comprehensive** (cover all major code paths, including success, failure, retry, ignored/no-op branches)
5. **Consistent** (shared event names, attribute schema, and severity conventions)

Do **not** treat this as a partial logging pass. The objective is to make the entire application operable from logs.

---

# Repository context

This is a Cloudflare Workers app using:

- Hono routes under `apps/api/src/routes/`
- Durable Objects under `apps/api/src/durable-objects/`
- shared libs under `apps/api/src/lib/`
- provider wrappers under `apps/api/src/services/`
- tail worker under `apps/tail/src/`

Important project constraints from `AGENTS.md`:

- Use the existing `Result<T, E>` style everywhere
- DO-to-DO communication uses RPC via `getStub()` from `apps/api/src/lib/durable-objects.ts`
- Errors are `TaggedError`s and should log `_tag` + `message`
- Use Zod at boundaries
- ISO timestamps only
- No raw SQL
- Be careful not to log secrets or unsafe payloads

# High-level deliverables

## 1. Replace the current logger with a context-aware structured logger

Primary file:

- `apps/api/src/lib/logger.ts`

The current logger is too thin. Replace/enhance it to support:

- `debug/info/warn/error`
- `child(context)` for inherited context
- stable top-level log shape
- error normalization helper
- timer helper / duration helper
- context merge semantics
- redaction/sanitization
- ability to emit machine-readable `event` names and human-readable `message`

### Required log shape

Every log line should be valid JSON and support the following fields:

```ts
{
  ts: string,                    // ISO8601 timestamp
  level: "debug" | "info" | "warn" | "error",
  event: string,                 // machine-readable event name
  message: string,               // human-readable summary
  component: string,             // route | durable_object | service | worker | tail | client | cache | analytics
  code_path?: string,            // file/function or route identifier

  request_id?: string,
  trace_id?: string,
  span_id?: string,
  parent_span_id?: string,

  route?: string,
  method?: string,
  path?: string,
  status_code?: number,
  duration_ms?: number,

  do_name?: string,
  do_id?: string,
  rpc_method?: string,

  provider?: "spotify" | "twitch",
  external_url?: string,
  external_status?: number,

  event_id?: string,
  message_id?: string,
  subscription_id?: string,
  subscription_type?: string,

  redemption_id?: string,
  reward_id?: string,
  saga_id?: string,
  stream_session_id?: string,

  user_id?: string,
  user_display_name?: string,
  track_id?: string,
  command?: string,

  outcome?: string,
  retryable?: boolean,
  attempt?: number,
  next_retry_at?: string,

  error_tag?: string,
  error_message?: string,
  error_stack?: string,

  ...additional structured attrs
}
```

### Logger API expectations

Provide a minimal ergonomic API such as:

- `logger.debug(message, attrs)`
- `logger.info(message, attrs)`
- `logger.warn(message, attrs)`
- `logger.error(message, attrs)`
- `logger.child(baseAttrs)`
- helper like `normalizeError(error)` or `toLogError(error)`
- helper like `startTimer()` returning duration

If useful, support a preferred call shape like:

```ts
logger.info("Request completed", {
  event: "http.request.completed",
  component: "route",
  route: "/api/now-playing",
  duration_ms: 23,
});
```

### Redaction requirements

The logger or helpers must ensure we **never log**:

- OAuth authorization `code`
- access tokens
- refresh tokens
- client secrets
- `Authorization` headers
- EventSub signature headers
- full webhook raw bodies
- full chat message body by default
- raw redemption `user_input` by default

Instead log safe summaries, e.g.:

- `code_present: true`
- `token_present: true`
- `message_length`
- `body_size_bytes`
- `input_length`
- `input_kind`
- `response_body_excerpt` only when safe and truncated

### Severity conventions

Use consistent severity:

- `debug`: noisy internal progress, cache hits, replay paths, per-step diagnostics
- `info`: successful domain transitions and meaningful milestones
- `warn`: degraded but recoverable, retries, ignored/stale/duplicate events, skipped branches
- `error`: hard failures, permanent errors, corrupted state, infrastructure failure

## 2. Add request-scoped logging middleware to the API worker

Primary file:

- `apps/api/src/index.ts`

Add Hono middleware that runs for **all requests**.

### Required behavior

Emit:

- `http.request.received`
- `http.request.completed`
- `http.request.failed`

For every route, including `/health`.

### Required attributes

On request received:

- `request_id` (generated UUID if none exists)
- `trace_id` (prefer upstream correlation if present; otherwise use request_id)
- `method`
- `path`
- `route` if available
- `query_keys` (not raw values unless harmless)
- `user_agent`
- `cf_ray` if available
- `client_ip_hash` or omit raw IP

On completion:

- `request_id`
- `trace_id`
- `method`
- `path`
- `route`
- `status_code`
- `duration_ms`

On failure:

- same as above plus `error_tag`, `error_message`, optional `error_stack`

### Context propagation

Store request logger / correlation IDs in Hono context so route handlers can derive child loggers.

## 5. Introduce cross-system correlation strategy

This is required so an operator can trace one event across the system.

### Correlation roots

Use these correlation IDs consistently:

- HTTP request: `request_id`
- EventSub webhook: `trace_id = twitch-eventsub-message-id`
- Song request saga: `trace_id = redemption.id`, `saga_id = redemption.id`
- Keyboard raffle saga: same pattern
- Event bus delivery: carry `event.id` and ideally `correlationId` / `causationId`
- Stream lifecycle transition: `stream_session_id`

### Action items

1. Propagate request logger/context into routes
2. For webhooks, seed logger from `messageId`
3. For sagas, create child loggers with `saga_id`, `redemption_id`, `user_id`, etc.
4. For EventBus events, extend event schemas if necessary to include `correlationId` and `causationId`
5. Ensure logs emitted from downstream handlers include the inherited IDs

If schema changes are made, update:

- `apps/api/src/durable-objects/schemas/event-bus-do.schema.ts`
- any helper creators such as `createSongRequestSuccessEvent`, `createRaffleRollEvent`, `createStreamOnlineEvent`, `createStreamOfflineEvent`

Backward compatibility should be considered if tests exist.
