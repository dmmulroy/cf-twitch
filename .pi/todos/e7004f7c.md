Root cause: Agent-backed Durable Object RPC calls were reaching production before the Agents runtime had a persisted instance name, causing `Attempting to read .name on _SongQueueDO before it was set` during SongQueueDO scheduling/refresh work. 

Fixes made:
- Updated `apps/api/src/lib/durable-objects.ts` so `getStub()` lazily calls `setName(resolvedId)` before the first RPC on Agent-backed stubs.
- Updated `apps/api/src/routes/webhooks.ts` to use `getStub()` for saga DOs instead of raw namespace `.get(...)`, so the same auto-naming path applies there too.
- Added regression test `apps/api/src/__tests__/durable-objects/get-stub-agent-name.test.ts` proving SongQueueDO can sync/schedule via `getStub()` without manual `setName()`.
- Simplified `keyboard-raffle-do.test.ts` to rely on `getStub()` auto-naming.
