## Goal

Evaluate and likely implement migrating `apps/api/src/durable-objects/stream-lifecycle-do.ts` from a plain `DurableObject<Env>` to Cloudflare's `Agent` base class (from the Agents SDK), using the Agent class as a better core operating model for Durable Objects **even though this object is not AI-related**.

This todo is intended as a full handoff for an agent with **no prior context** from the original discussion.

---

## Primary recommendation from prior research

**Best first candidate to migrate: `StreamLifecycleDO`**

### Why this DO is the strongest first migration candidate

1. **It already matches the Agent lifecycle model well**
   - It has event-style lifecycle methods: `onStreamOnline()`, `onStreamOffline()`
   - It exposes a custom HTTP-ish API through a `fetch()` path switch
   - It runs periodic background work via `alarm()`
   - It maintains a small current-state model plus historical data
   - It broadcasts websocket-style notifications manually

   This maps naturally to Agent primitives:
   - `onStart(props?)`
   - `onRequest(request)`
   - `schedule()` / `scheduleEvery()`
   - `this.state` / `setState()`
   - `onConnect()` / `broadcast()`

2. **It fixes an active project violation immediately**
   The repo instructions explicitly call out `StreamLifecycleDO` as violating the preferred pattern:
   - it has a `fetch()` switch router and “should expose RPC methods directly”

   Adopting `Agent` gives a clean replacement:
   - custom `fetch()` switch → `onRequest()`
   - keep explicit RPC methods where useful
   - align this DO with a framework-native model instead of hand-rolling server behavior

3. **Its state shape is a strong fit for `Agent.state`**
   Natural split:
   - **Current/live state** → Agent state
     - `isLive`
     - `startedAt`
     - `endedAt`
     - `peakViewerCount`
     - `streamSessionId`
   - **Historical/query-heavy data** → keep in Drizzle/SQLite
     - `viewerSnapshots`

4. **It gets value from multiple Agent features at once**
   Unlike simpler DOs, this migration demonstrates actual value:
   - `onRequest()` instead of custom routing
   - `this.state` / `setState()` for current stream state
   - optional automatic state sync to clients
   - `broadcast()` instead of manual websocket fanout
   - `scheduleEvery()` / `schedule()` instead of raw `setAlarm()`

5. **It is less risky than the more complex infrastructure DOs**
   Not recommended as the first migration:
   - `event-bus-do.ts` → infrastructure-critical retry/DLQ semantics
   - `song-queue-do.ts` → very complex sync/reconciliation logic
   - `song-request-saga-do.ts`, `keyboard-raffle-saga-do.ts` → already orchestration-heavy
   - `achievements-do.ts` → broad business-rule surface

   `StreamLifecycleDO` is important but still bounded enough for a first framework migration.

## Secondary prioritization notes

If the project wants a staged adoption path, recommended order:

1. `StreamLifecycleDO`
2. `CommandsDO` (safe/boring validation that Agent also works for non-realtime DOs)
3. `SpotifyTokenDO`, `TwitchTokenDO`
4. `KeyboardRaffleDO`
5. `EventBusDO`
6. Later: `SongQueueDO`, `AchievementsDO`, saga DOs

### Why not `CommandsDO` first?
It is safer, but it doesn't prove much of the **value** of Agents because it does not use alarms, websockets, state sync, or richer lifecycle primitives.

### Why not token DOs first?
Good later candidates, but their refresh/alarm/coalescing behavior is subtle and they benefit from fewer Agent features than `StreamLifecycleDO`.

## Cloudflare Agents SDK research summary

### High-level conclusion
Cloudflare's `Agent` base class is best understood as:

> **an opinionated stateful application framework built on Durable Objects**

Layering:
- `DurableObject`
- `partyserver` `Server`
- `agents` `Agent`

### What Agent adds on top of plain Durable Objects

#### Lifecycle and routing
- `onStart(props?)`
- `onRequest(request)`
- `onConnect(connection, ctx)`
- `onMessage(connection, message)`
- `onClose(connection, code, reason, wasClean)`
- `onError(...)`

#### Built-in state model
- `initialState`
- `this.state`
- `this.setState()`
- `validateStateChange(nextState, source)`
- `onStateChanged(state, source)`

Important behavior:
- state persisted in internal SQLite table `cf_agents_state`
- state lazily loaded
- state broadcasts to connected protocol-enabled clients automatically
- `validateStateChange()` is synchronous and gating
- `onStateChanged()` runs **after** persist + broadcast

#### WebSocket/client RPC
- `@callable()`
- `@callable({ streaming: true })`
- `StreamingResponse`

#### Background execution
- `queue()` / dequeue helpers
- `schedule()`

- `broadcast()`

## Current StreamLifecycleDO design summary

### What it does today
- owns singleton stream lifecycle state
- tracks whether stream is live
- stores `startedAt`, `endedAt`, `peakViewerCount`
- periodically polls Twitch viewer count every 60s while live
- stores viewer snapshots historically
- notifies token DOs when stream transitions online/offline
- publishes stream events to `EventBusDO`
- exposes a custom `fetch()` path router:
  - `/stream-online`
  - `/stream-offline`
  - `/record-viewer-count`
  - `/state`
  - `/history`
  - `/is-live`
  - `/ping`
- manually broadcasts websocket messages and closes websocket clients on offline

### Current design issues
- custom `fetch()` router inside DO is a project anti-pattern here
- `isLive` and `streamSessionId` are partly in-memory and can drift across hibernation
- websocket broadcasting is manual
- alarm scheduling is raw/manual
- state split is not clearly modeled as “current state vs history”

## Explicit migration design for porting StreamLifecycleDO to Agent

## Migration objective
Port `StreamLifecycleDO` from:
- `class _StreamLifecycleDO extends DurableObject<Env>`

to something like:
- `class StreamLifecycleDO extends Agent<Env, StreamLifecycleAgentState>`

while:
- keeping Drizzle for historical/queryable data
- preserving existing external behavior where practical
- simplifying lifecycle, scheduling, and current-state handling
- minimizing downstream changes

---

## Proposed target architecture

### 1. Introduce explicit Agent state type
Create a dedicated current-state type, e.g.:

```ts
interface StreamLifecycleAgentState {
  isLive: boolean;
  startedAt: string | null;
  endedAt: string | null;
  peakViewerCount: number;
  streamSessionId: string | null;
}
```

### 2. Use Agent state for the singleton live-state model
Move these concerns into `Agent.state`:
- `isLive`
- `startedAt`
- `endedAt`
- `peakViewerCount`
- `streamSessionId`

This is the most important design change.

### 3. Keep historical data in Drizzle/SQLite
Keep using Drizzle schema + migrations for:
- `viewerSnapshots`
- any long-lived historical/query needs

### 4. Remove manual in-memory authoritative state
Eliminate or sharply reduce these in-memory fields:
- `private isLive = false`
- `private streamSessionId: string | null = null`

The persisted Agent state should become the source of truth.

### 5. Replace raw `alarm()` pattern with Agent scheduling
Preferred design:
- when stream goes online, create/start an interval schedule for viewer polling
- when stream goes offline, cancel that schedule

Use either:
- `scheduleEvery(60, "pollViewerCountTick")`

or, if exact compatibility requires more control:
- one-shot `schedule(60, ...)` chained manually

**Recommendation:** prefer `scheduleEvery()` for clarity unless tests/behavior show you need custom rearming semantics.

### 6. Replace custom `fetch()` switch with `onRequest()`
Move path-based handling into Agent `onRequest()`.

Do **not** recreate a giant anti-pattern switch if avoidable, but a bounded switch inside `onRequest()` is acceptable if preserving public behavior is necessary.

### 7. Keep internal methods callable by worker-side RPC as regular public methods
Worker/DO internal callers do not need `@callable()`.
They can continue to use internal RPC methods via `getStub()`.

### 8. Do not over-adopt client sync on day 1
Agent auto state sync is available, but this migration should not require building a full client-side Agent protocol consumer unless it is already useful.

### 9. Prefer `broadcast()` over manual websocket loops if direct websocket connections remain needed
If this DO still has direct websocket clients after migration:
- use Agent connection management + `broadcast()`
- avoid `ctx.getWebSockets()` manual fanout

If there are no real websocket clients today, remove dead/manual websocket behavior rather than preserving it blindly.

## Detailed target behavior for each method

### `onStart()`
Responsibilities:
- initialize Drizzle
- run migrations if necessary
- ensure historical schema exists
- ensure Agent state is coherent
- during transition, optionally backfill Agent state from legacy singleton table

Preferred end-state:
- `Agent.state` is the authoritative current-state source
- no dependence on a separate `streamState` DB singleton for live fields

### `onStreamOnline(eventTimestamp?)`
Behavior:
1. resolve authoritative timestamp with current validation logic
2. compare against current state to reject stale or duplicate online events
3. generate new `streamSessionId`
4. update Agent state:
   - `isLive = true`
   - `startedAt = now`
   - `endedAt = null`
   - `peakViewerCount = 0`
   - `streamSessionId = new UUID`
5. optionally mirror to DB singleton only during transition period
6. notify token DOs online
7. publish `stream_online` event
8. start viewer polling schedule if not already running
9. optionally broadcast stream-online message to connected clients

### `onStreamOffline(eventTimestamp?)`
Behavior:
1. resolve timestamp
2. reject stale or duplicate offline events
3. update Agent state:
   - `isLive = false`
   - `endedAt = now`
   - preserve `startedAt`
   - preserve `peakViewerCount`
   - keep `streamSessionId` long enough to publish offline event, then clear it
4. notify token DOs offline
5. publish `stream_offline` event
6. cancel viewer polling schedule
7. optionally broadcast stream-offline message
8. optionally close active connections if that behavior is still wanted

### `recordViewerCount(count)`
Behavior:
1. insert a snapshot row into `viewerSnapshots`
2. if `count > this.state.peakViewerCount`, update Agent state peak
3. no need for a separate in-memory peak cache

### `pollViewerCountTick()`
New scheduled callback method, replacing raw alarm logic.
Behavior:
1. if `!this.state.isLive`, exit early
2. call `pollViewerCount()` via `TwitchService`
3. if count returned, call `recordViewerCount(count)`
4. if using `scheduleEvery()`, nothing else needed
5. if using chained one-shot schedules, reschedule only when live

### `getStreamState()`
Prefer returning from Agent state rather than separate DB singleton.
If route/API consumers expect an existing DB-shaped type, either:
- adapt Agent state to that type, or
- keep a narrow compatibility shape

### `getViewerHistory(since?, until?)`
Keep this Drizzle-backed.
This is historical/query data, not Agent state.

### `getIsLive()`
Return `this.state.isLive`.

### `ping()`
Can remain trivial.

Use Agent state for:
- `isLive`

## Scheduling design recommendation

### Preferred design: `scheduleEvery()`
Use an interval schedule while the stream is live.

Example shape:
- field or lookup to keep the schedule id, or infer from `getSchedules()` if needed
- start it on `onStreamOnline()`
- cancel it on `onStreamOffline()`

Possible callback name:
- `pollViewerCountTick`

### Why this is better than raw alarm
- clearer intent
- no manual `setAlarm()` bookkeeping
- aligns with Agent's lifecycle and scheduling model
- easier future consistency with other Agent-based DOs

### Caveat
If exact “one alarm at a time only while live” semantics matter and interval overlap behavior becomes a problem, fall back to explicit `schedule(60, ...)` chaining. But start with `scheduleEvery()`.

## Error handling design recommendation

The project uses `Result<T, E>` extensively.
Do **not** throw that away because you are adopting `Agent`.

Recommended pattern:
- keep worker/DO-facing methods returning `Result` where they already do
- keep Zod validation at request boundaries
- use Agent APIs for lifecycle/scheduling/state, but preserve existing error-as-value style for domain operations

`Agent` does not require a throw-heavy architecture.
