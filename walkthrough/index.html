The one-line summary

pi-share-hf turns local pi coding-agent sessions into a publishable Hugging Face dataset, but only after layered safety checks: exact secret redaction, deny-pattern filtering, TruffleHog scanning, LLM review, and manual rejection support.

What we optimized for in cf-twitch

This repo has real OAuth credentials and operational context for Twitch and Spotify, so the integration is deliberately safe by default: local workspace, local helper files, --no-images, explicit project context, and generated secret lists.

✅
The outcome: using the feature is now a small set of repeatable commands instead of a long, easy-to-get-wrong CLI incantation.

Background

How pi-share-hf works as a pipeline

Think of it as a staged filter from raw session traces to an uploadable public dataset.

Raw sessions do not go straight to Hugging Face; every stage narrows the candidate set.

High-precision redaction first

The first layer only knows exact values. That is why we generate a secret list from local env files instead of guessing with broad regexes.

Semantic review later

The LLM review is there to catch things that are not token-shaped secrets: mixed-project sessions, private topics, or sensitive operational context that survives literal redaction.

The problem

Why this repo needed a cautious integration

The default pi-share-hf workflow is good, but cf-twitch has specific risk factors that justified wrapper scripts and stricter defaults.

ConcernGeneric repocf-twitch realityDesign response
Secrets location maybe in shell rc or provider settings real Twitch / Spotify secrets in .envrc force --env-file .envrc + generated secrets
Project context README may be enough no root README; architecture context lives in docs pass 4 explicit context files to review
Images sometimes useful to preserve screenshots may include dashboards / callbacks default to --no-images
Human ergonomics long CLI commands easy to omit flags or use wrong defaults npm scripts encode the repo-safe path
⚠️
The key risk: this repo touches OAuth, webhooks, broadcaster identity, admin/setup secrets, and local operational details. A “just publish my sessions” workflow would be too fragile.
The integration is mostly about closing repo-specific safety gaps and turning them into defaults.

Init path

How pnpm share:init bootstraps the workspace

The init wrapper exists so the first-time setup produces the right local shape for this repo every time.

1

Create .pi/hf-private/ if needed

We make sure the local helper directory exists before anything else. This is where local-only inputs such as deny.txt live.

2

Seed a starter deny list

If .pi/hf-private/deny.txt does not exist, the script writes defaults for your home path and common tunnel hosts.

3

Resolve the dataset repo target

Default repo is dmmulroy/cf-twitch-pi-sessions, but you can override it with PI_SHARE_HF_REPO and optionally PI_SHARE_HF_ORGANIZATION.

4

Invoke pi-share-hf with repo-safe defaults

The wrapper calls pi-share-hf init --workspace .pi/hf-sessions --repo ... --no-images, encoding the “no images” decision into the normal path.

// scripts/share-hf-init.mjs
const repo = process.env.PI_SHARE_HF_REPO ?? "dmmulroy/cf-twitch-pi-sessions";
const args = [
  "init",
  "--workspace", workspaceDir,
  "--repo", repo,
  "--no-images",   // disable screenshot/image publishing by default
];

if (organization) {
  args.push("--organization", organization);
}

const result = spawnSync("pi-share-hf", args, { stdio: "inherit" });
✅
Design win: after this script exists, “how do I initialize it safely?” becomes a one-command answer.

Secrets path

How we generate the exact-secret redaction input

This is the most repo-specific part of the integration because the project keeps real credentials in local env files.

We are not asking pi-share-hf to guess our secrets; we are feeding it an exact, current list of real values to redact.

Why exact values matter

Exact-value redaction is high precision. If the real Twitch client secret appears in a session and that exact value is in secrets.txt, it will be redacted with very low false-positive risk.

Why the script strips quotes and dedupes

Local env files can contain quoted values or duplicates. The script normalizes those cases so the resulting file is a clean list of exact secret strings.

// scripts/share-hf-secrets.mjs
const envFiles = [resolve(rootDir, ".envrc"), resolve(rootDir, ".dev.vars")];
const envVarNames = [
  "OPENAI_API_KEY",
  "ANTHROPIC_API_KEY",
  "HF_TOKEN",
  "GOOGLE_API_KEY",
  "OPENROUTER_API_KEY",
];

const values = new Set();
for (const filePath of envFiles) {
  collectFromEnvFile(filePath, values);   // parse local repo env files
}
for (const name of envVarNames) {
  if (process.env[name]) values.add(process.env[name]);
}
writeFileSync(outputFile, sortedValues.join("\n") + "\n", "utf8");

Usage path

How the everyday commands map to the underlying pipeline

The npm scripts are the human-facing API of the integration.

collect

Regenerates secrets.txt, then runs pi-share-hf collect with repo-specific flags: workspace, env file, deny list, and context files.

review

Reruns the semantic review stage against already-redacted sessions. Good for updated deny rules or changed project context.

upload

Pushes only the sessions currently marked uploadable. The dry-run variant lets you inspect counts before sending anything to HF.

// package.json
"share:collect": "pnpm share:secrets && pi-share-hf collect --workspace .pi/hf-sessions --env-file .envrc --secret .pi/hf-private/secrets.txt --deny .pi/hf-private/deny.txt AGENTS.md docs/prd.md docs/cf-prototype-architecture.md docs/effect-twitch-architecture.md",
"share:review": "pnpm share:secrets && pi-share-hf review --workspace .pi/hf-sessions --deny .pi/hf-private/deny.txt AGENTS.md docs/prd.md docs/cf-prototype-architecture.md docs/effect-twitch-architecture.md",
"share:list": "pi-share-hf list --workspace .pi/hf-sessions --uploadable",
"share:upload:dry": "pi-share-hf upload --workspace .pi/hf-sessions --dry-run",
"share:upload": "pi-share-hf upload --workspace .pi/hf-sessions"
1

pnpm share:collect

Update exact secrets, collect changed sessions, redact, filter, scan, and review.

2

pnpm share:list

Inspect the current uploadable set. This is the “what would go public right now?” question.

3

Optional grep / manual reject

Search for risky terms or reject anything that still feels wrong, even if it technically passed automated checks.

4

pnpm share:upload:dry → pnpm share:upload

Dry-run first, then do the real upload only when satisfied with the candidate set.

Files + policy

The repo conventions we established

These choices are the real integration: they turn a standalone tool into a team-usable project workflow.

Gitignore policy

# .gitignore
.pi/hf-sessions/
.pi/hf-private/

This ensures the share workspace and helper files remain local. The repo gets the workflow, not the generated private state.

Context policy

  • AGENTS.md
  • docs/prd.md
  • docs/cf-prototype-architecture.md
  • docs/effect-twitch-architecture.md

Because there is no root README, the review model gets an explicit bundle of architectural context instead of relying on implicit defaults.

✅ Strong default

Images are off by default. This is exactly the right bias for a repo that may contain screenshots of OAuth flows, callback hosts, or dashboards.

⚠️ Still needs human review

Deny lists evolve. The starter deny.txt is just a beginning; you should add private names, domains, and project references over time.

💬 Nice ergonomics

Scripts encode intent. Team members do not have to remember the exact workspace path or context-file bundle; the repo already knows.

Takeaways

How to think about this integration going forward

The most useful mental model is “repo policy + safety inputs + publish commands.”

What pi-share-hf owns

  • session collection and incremental state
  • exact-value redaction
  • TruffleHog scanning
  • LLM review and upload gating
  • uploading the final dataset

What this repo now owns

  • where workspace state lives
  • which env file is authoritative
  • which context files define the project
  • what safety defaults are enforced
  • how humans invoke the workflow
💡
Best mental model: pi-share-hf is the generic engine; the scripts and policy files we added are the cf-twitch adapter.

Recommended day-to-day sequence

pnpm share:init        # once per local workspace
pnpm share:collect     # regenerate secrets + process changed sessions
pnpm share:list        # inspect uploadable sessions
pnpm share:upload:dry  # check what would be pushed
pnpm share:upload      # publish when satisfied
✅
Bottom line: the integration is complete enough that you can learn the workflow from the repo itself, not from tribal memory or ad hoc shell history.
